# Threat Model: FD-004 — Cross-cluster secret injection via remote OpenBao

> Generated by `/fd-threat-model` on 2026-04-18.
> Based on: `.forgia/fd/FD-004-cross-cluster-secret-injection.md`

## 1. Assets

| Asset | Type | Location | Sensitivity |
|-------|------|----------|-------------|
| AppRole role_id | Credential | K8s Secret on consumer cluster | **High** — identifies the auth role; static, long-lived |
| AppRole secret_id | Credential | K8s Secret on consumer cluster | **Critical** — combined with role_id grants Vault token; must be rotated |
| Vault token (issued by AppRole login) | Credential | In-memory (vault-env process) | **High** — grants access to tenant's KV secrets; short TTL mitigates |
| Tenant KV secrets | Data | OpenBao on baobank | **High** — application credentials (DB passwords, API keys) |
| Docker bridge network | Infrastructure | Docker daemon | **Medium** — both Kind clusters share it; no encryption |
| NodePort / Docker IP exposure | Infrastructure | baobank Kind node | **High** — exposes OpenBao to Docker network; reachable by any container |
| Consumer cluster kubeconfig | Credential | Local filesystem | **High** — admin access to consumer cluster |

## 2. Threat Actors

| Actor | Motivation | Access Level | Relevance to FD |
|-------|-----------|--------------|-----------------|
| Container on Docker network | Reach OpenBao exposed via NodePort | Any Docker container on the bridge | NodePort on baobank is reachable by ALL containers on the Docker bridge — not just the consumer cluster |
| Compromised consumer pod | Steal AppRole credentials from K8s Secret or env vars | Pod-level access on consumer cluster | If pod is compromised, attacker gets the secret_id and can mint Vault tokens |
| Tenant user on baobank | Escalate from own OpenBao to another tenant via cross-cluster path | Capsule tenant user | If AppRole is misconfigured, a tenant user could create a role_id/secret_id that accesses paths beyond their scope |
| Network sniffer on Docker bridge | Capture plaintext Vault API traffic | Passive network access | No TLS = secrets in plaintext on the wire |

## 4. Recommendations for SDDs

1. **SDD-001** (Consumer cluster + exposure): Document that NodePort exposes OpenBao to the entire Docker bridge network — not just the consumer cluster. For production, use Ingress with TLS and IP allowlist. In the BATS test, verify OpenBao is reachable only from the expected consumer pod.

2. **SDD-002** (AppRole provisioning): Set `secret_id_ttl=24h` and `token_ttl=1h` on the AppRole. Policy must be scoped to `secret/data/acme/*` (not `secret/*`). BATS: verify the AppRole's policy path is tenant-scoped. Verify issued token has TTL <= 1h.

3. **SDD-003** (vault-env deployment): Pin vault-env image by tag. Set `VAULT_CLIENT_TIMEOUT=10s`. BATS: verify running image matches expected tag. Verify the pod starts with correct env var value.

4. **SDD-004** (BATS + wiring): Include security tests: `auth can-i get secrets -n <secret-ns> --as default` on consumer cluster should fail for non-vault SAs. Verify AppRole token has limited TTL. Verify plaintext-on-wire risk is documented (no TLS assertion).
