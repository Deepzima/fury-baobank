# Threat Model: FD-003 — OpenBao secret management via Bank-Vaults operator (per-tenant)

> Generated by `/fd-threat-model` on 2026-04-17 (v2 — per-tenant architecture).

## 1. Assets

| Asset | Type | Location | Sensitivity |
|-------|------|----------|-------------|
| Per-tenant unseal keys (N Secrets, one per tenant) | Credential | K8s Secret in each tenant namespace | **Critical** — compromise of one = full access to that tenant's secrets only (blast radius contained) |
| Per-tenant root token (N Secrets) | Credential | K8s Secret in each tenant namespace | **Critical** — root access to one tenant's Vault |
| Tenant secret data (KV-v2, per-instance) | Data | OpenBao encrypted Raft storage (PV per tenant) | **High** — application credentials, DB passwords, API keys |
| Vault CR spec (per-tenant) | Data | Tenant namespace (applied by platform operator or onboarding controller) | **High** — defines policies, auth config; modification = policy bypass for that tenant |
| Kubernetes ServiceAccount tokens (per-tenant) | Credential | K8s API (auto-mounted in tenant pods) | **High** — used for OpenBao Kubernetes auth; scoped to tenant's Vault role |
| Bank-Vaults webhook TLS cert | Credential | K8s Secret in `bank-vaults-system` | **High** — MITM the webhook = intercept/modify secret injection for ALL tenants |
| Bank-Vaults operator RBAC (cross-namespace) | Infrastructure | ClusterRole/ClusterRoleBinding | **Critical** — operator has broad permissions: creates StatefulSets, reads/writes Secrets across ALL tenant namespaces |
| Per-tenant OpenBao audit log | Data | OpenBao file backend or stdout per instance | **Medium** — per-tenant accountability; tampering hides who accessed what |
| Per-tenant Raft storage PVs | Infrastructure | PersistentVolumes on Kind nodes (one per tenant) | **High** — physical access to PV = encrypted data (needs unseal keys) |
| Capsule tenant boundary + NetworkPolicy | Infrastructure | Capsule Tenant CR + namespace labels | **Critical** — prevents cross-tenant network access to OpenBao endpoints |
| Bank-Vaults webhook MutatingWebhookConfiguration | Infrastructure | Cluster-scoped K8s resource | **High** — modification = redirect admission reviews for ALL tenant pods |
| Per-tenant OpenBao listener TLS | Credential | TLS cert/key per instance (cert-manager or operator-managed) | **High** — without TLS, MITM on secret read/write within tenant scope |
| Tenant onboarding flow (Vault CR creation) | Infrastructure | K8s API (who can create Vault CRs in tenant namespaces) | **High** — unauthorized Vault CR creation = rogue OpenBao instance consuming resources or impersonating a tenant |

## 2. Threat Actors

| Actor | Motivation | Access Level | Relevance to FD |
|-------|-----------|--------------|-----------------|
| Malicious tenant user (tenant-A) | Access tenant-B's secrets, or escalate beyond own Vault | Authenticated, limited to own Capsule namespace | Core isolation test — per-tenant architecture means the primary boundary is namespace + network, not Vault policy |
| Compromised tenant application pod | Exfiltrate secrets from own or other tenant's OpenBao | Authenticated via own SA, has injected secrets in env | If pod is compromised, attacker gets env var secrets. With per-tenant model, they can only reach their own OpenBao — network isolation is the barrier to cross-tenant |
| Rogue tenant creating unauthorized Vault CRs | Spin up extra OpenBao instances to exhaust cluster resources or create a backdoor endpoint | Authenticated, Capsule namespace owner | If Capsule allows the tenant user to create `Vault` CRs (Bank-Vaults CRD), they could deploy rogue instances |
| Cluster admin with stolen kubeconfig | Full cluster access | cluster-admin RBAC | Can read ALL unseal keys across ALL tenants — N Secrets instead of 1. Blast radius = all tenants. Lab-only accepted risk |
| Compromised Bank-Vaults operator | Access all tenant Vault instances simultaneously | Runs as operator with cross-namespace RBAC | **Highest impact actor in per-tenant model** — operator has unseal keys for every tenant. Single point of trust for the entire platform |
| Supply chain (compromised OpenBao image) | Backdoor all tenant Vault instances | Runs as OpenBao in every tenant namespace | All N instances use the same image — supply chain attack compromises every tenant simultaneously |
| Unauthenticated network attacker (intra-cluster) | Probe or intercept OpenBao API traffic between tenants | Network access within the cluster (pod-to-pod) | Per-tenant model creates N OpenBao Services — each is a potential target. Without NetworkPolicy, any pod can reach any tenant's OpenBao |

## 3. STRIDE Analysis

### Bank-Vaults Operator (cross-namespace controller)

| Threat | Category | Risk | Mitigation |
|--------|----------|------|------------|
| Operator has RBAC to read/write Secrets in ALL tenant namespaces — compromise = all unseal keys leaked | **I** (Information Disclosure) | **Critical** | Pin operator image by digest. Restrict operator RBAC to only the verbs needed (get/create/update on Secrets, not list across cluster). Monitor operator pod for unexpected restarts. |
| Operator modifies a tenant's Vault CR to inject rogue policies or change the OpenBao image | **T** (Tampering) | High | Operator should NOT have write access to Vault CRs (it only reads them). Verify ClusterRole does not grant `update` on `vaults.vault.banzaicloud.com`. |
| Operator reconciles N instances — a slow reconciliation loop could leave tenant instances sealed for minutes after restart | **D** (Denial of Service) | Medium | Monitor reconciliation latency. For the lab with 2 tenants this is not a concern; at scale (100+ tenants) the operator may need sharding or multiple replicas. |
| Operator logs may contain init output across multiple tenants (root tokens, unseal keys) | **I** (Information Disclosure) | High | Set operator log level to `info`. BATS test: grep operator logs post-init for known token/key patterns — must find none. |
| No audit trail of which tenant's Vault CR the operator reconciled and when | **R** (Repudiation) | Medium | K8s audit log covers CR watch/get events. Operator logs should include tenant name per reconciliation cycle. |
| No threats identified | **S** (Spoofing) | — | Operator authenticates via ServiceAccount token to K8s API. No external identity to spoof. |
| Operator's cross-namespace RBAC is the broadest permission in the cluster (broader than any single tenant) | **E** (Elevation of Privilege) | High | Audit operator ClusterRole in BATS: verify no `["*"]` verbs, no `["*"]` resources. Document this as the "trust root" of the platform — the operator SA must be as protected as cluster-admin. |

### Bank-Vaults Webhook (shared across all tenants)

| Threat | Category | Risk | Mitigation |
|--------|----------|------|------------|
| Webhook is shared — it processes AdmissionReviews for ALL tenant namespaces. A spoofed webhook endpoint hijacks injection for every tenant | **S** (Spoofing) | **Critical** | BATS test: assert webhook `clientConfig.service.namespace` = `bank-vaults-system`. Lock down RBAC on MutatingWebhookConfiguration (only cluster-admin can modify). |
| Webhook authenticates to each tenant's OpenBao to read secrets. If the webhook's own Vault token cache is compromised, all tenant secrets are exposed | **I** (Information Disclosure) | **Critical** | Verify webhook authenticates using the target pod's ServiceAccount, not a global SA. Each injection is scoped to the tenant's Vault endpoint and the pod's SA role. |
| Webhook injects secrets as plaintext env vars — any process in the pod reads them via `/proc/1/environ` | **I** (Information Disclosure) | Medium | Accepted trade-off for the lab. Document: production should consider Vault Agent sidecar with tmpfs. |
| Webhook TLS cert may have the same `isCA` issue as Capsule (FD-002) | **T** (Tampering) | High | Verify cert-manager Certificate has `isCA: true` if self-signed. Apply same kustomize patch pattern as FD-002 if needed. BATS test: inspect caBundle. |
| Webhook `failurePolicy: Fail` blocks ALL pod creation across ALL tenants if webhook is down | **D** (Denial of Service) | **Critical** | Webhook must have replicas >= 2 for HA. Exempt system namespaces. `timeoutSeconds: 5`. BATS test: verify readyReplicas >= 2. |
| Webhook reads a secret and injects it — no per-injection log | **R** (Repudiation) | Medium | Enable OpenBao audit backend on each tenant instance. The Vault audit log records every secret read. Webhook logs (info level) should show injection events per pod. |
| Webhook could inject secrets from the wrong tenant's OpenBao if the vault-addr annotation is crafted to point at another tenant's Service | **E** (Elevation of Privilege) | **Critical** | Vault Kubernetes auth role is bound to `bound_service_account_namespaces`. Even if annotation points to tenant-B's OpenBao, the SA from tenant-A will fail authentication (namespace mismatch). BATS test: explicitly verify this. Additionally, Capsule NetworkPolicy should block cross-tenant network access. |

### Per-Tenant OpenBao Instance

| Threat | Category | Risk | Mitigation |
|--------|----------|------|------------|
| A pod in tenant-A namespace reaches tenant-B's OpenBao Service via direct ClusterIP | **E** (Elevation of Privilege) | **Critical** | Capsule must enforce NetworkPolicy per-tenant: deny egress to other tenant namespaces. Cilium can enforce this at L3/L4. BATS test: pod in tenant-alpha attempts `curl reevo-ob-id-beta.bats-tenant-beta:8200` — must fail (connection refused or timeout). |
| Tenant-A's SA authenticates against tenant-B's OpenBao (if network allows) | **S** (Spoofing) | High | Vault Kubernetes auth role: `bound_service_account_namespaces: ["bats-tenant-alpha"]`. SA from `bats-tenant-beta` fails auth. BATS test: explicit cross-tenant auth attempt → 403. |
| Unseal keys Secret in tenant namespace readable by tenant user (if Capsule grants `get secrets` in own namespace) | **I** (Information Disclosure) | **Critical** | Capsule tenant owners typically get admin-like access to their namespace. The unseal keys Secret MUST be protected. Options: (a) operator stores keys in a different namespace, (b) use K8s RBAC to deny tenant user access to the specific Secret name, (c) encrypt keys with a KMS (not available in Kind). For the lab: document the risk. BATS test: verify the Secret exists and note this as a production concern. |
| OpenBao API without TLS — secrets plaintext on the wire within tenant namespace | **I** (Information Disclosure) | High | Enable TLS on the OpenBao listener via Vault CR config. BATS test: `vault status -address=https://...` succeeds. |
| Tenant user deletes their own Vault CR → operator deletes StatefulSet → data loss | **D** (Denial of Service) | Medium | If tenant user has delete on `Vault` CRs in their namespace, they can destroy their own instance. Acceptable for "tenant-owned" model (they own it, they can break it). For production: add finalizers or restrict delete to platform operator. |
| No audit backend enabled by default — tenant operations unaccountable | **R** (Repudiation) | High | Enable `file` audit backend in the Vault CR template. BATS test: `vault audit list` on each instance shows enabled backend. |
| Tenant deploys a malicious pod that brute-forces Vault auth tokens | **D** (Denial of Service) | Low | Vault has built-in rate limiting and auth attempt throttling. Capsule ResourceQuota limits pod count. Low risk for a lab. |
| All N tenant instances use the same OpenBao image — supply chain attack compromises all simultaneously | **T** (Tampering) | High | Pin image by tag (ideally digest). BATS test: verify running image across all instances matches expected value. Consider image scanning in CI. |

## 4. Recommendations for SDDs

1. **SDD-001** (Bank-Vaults operator plugin): Pin operator image by tag (digest in prod). Set `logLevel: info`. Audit ClusterRole: must NOT have `["*"]` verbs or resources — document it as the platform trust root. Verify operator does not log unseal keys or root tokens (grep operator logs in BATS post-init).

2. **SDD-002** (Bank-Vaults webhook plugin): Set `failurePolicy: Fail`, `timeoutSeconds: 5`, `replicas: 2`. Verify cert-manager Certificate — apply `isCA: true` patch if needed (same FD-002 pattern). Exempt system namespaces from webhook scope. BATS: assert webhook service namespace = `bank-vaults-system`, assert no rogue webhook. **Critical**: verify webhook authenticates using the TARGET pod's ServiceAccount, not its own global SA.

3. **SDD-003** (Per-tenant OpenBao provisioning): Enable TLS on OpenBao listener. Enable `file` audit backend in Vault CR template. Pin OpenBao image `ghcr.io/openbao/openbao:2.1.0`. Kubernetes auth roles: `bound_service_account_namespaces` MUST match the tenant namespace exactly (no wildcards). **Critical**: inject a default-deny NetworkPolicy into each tenant namespace (via Capsule `GlobalTenantResources` or directly in the Vault CR kustomize) — this is the primary cross-tenant isolation boundary. RBAC: restrict `create/update/delete` on `vaults.vault.banzaicloud.com` to cluster-admin only (tenant users must NOT self-provision).

4. **SDD-004** (Tenant isolation BATS): Must include these critical security tests:
   - Cross-tenant network connectivity: `curl reevo-ob-id-beta.bats-tenant-beta:8200` from tenant-alpha pod → must fail (timeout/refused)
   - Cross-tenant auth: SA from tenant-alpha authenticates against tenant-beta's OpenBao → must fail 403
   - Unseal keys Secret: `auth can-i get secrets -n bats-tenant-beta --as alpha-user` → "no"
   - Vault CR creation: `auth can-i create vaults.vault.banzaicloud.com -n bats-tenant-alpha --as tenant-user` → "no"
   - Webhook spoofing: all Bank-Vaults webhook `clientConfig.service.namespace` = `bank-vaults-system`
   - Webhook HA: readyReplicas >= 2
   - All webhook timeoutSeconds <= 5
   - Per-instance TLS: `vault status -address=https://...` succeeds
   - Per-instance audit: `vault audit list` shows enabled backend
   - Image verification: running OpenBao image matches expected tag across all instances
   - Operator RBAC: ClusterRole has no `["*"]` verbs
   - **Cross-annotation injection**: pod in tenant-alpha with `vault-addr` pointing to tenant-beta's OpenBao → injection must fail (SA auth failure)

5. **SDD-005** (Integration wiring): Verify kapp ordering (operator → webhook → ready before tenant CRs applied). Context guard. Validate total test count = 38 + N on fresh cluster.
