# Threat Model: FD-006 — HSM-backed unseal + etcd encryption via Transit

> Generated by `/fd-threat-model` on 2026-04-18.
> Based on: `.forgia/fd/FD-006-hsm-unseal-etcd-encryption.md`

## 1. Assets

| Asset | Type | Location | Sensitivity |
|-------|------|----------|-------------|
| HSM wrapping key ("vault-unseal") | Credential | softhsm-kube token store (PV) | **Critical** — master key that encrypts ALL tenant unseal keys. Compromise = all tenants' Vaults exposed. |
| HSM PIN / slot credentials | Credential | Environment variable or K8s Secret in softhsm-system | **Critical** — grants access to the HSM slot; combined with network access to softhsm-kube = key extraction possible |
| HSM-encrypted unseal key blob | Data | K8s Secret in tenant namespace | **Medium** — encrypted, useless without HSM access (upgrade from Critical in FD-003) |
| Transit encryption key ("etcd-kms") | Credential | OpenBao Transit engine internal storage | **Critical** — encrypts all K8s Secrets in etcd. Compromise + etcd dump = all Secrets decrypted |
| Transit DEK (Data Encryption Key) | Credential | In-memory in vault-kubernetes-kms process | **High** — ephemeral, used for envelope encryption. Cached per KMS v2 spec. |
| encryption-config.yaml | Infrastructure | Kind control-plane node filesystem | **High** — misconfiguration = Secrets written in plaintext to etcd, or apiserver fails to start |
| etcd raw data | Data | etcd data directory on control-plane | **High** — contains all K8s state; with KMS v2 active, Secrets are encrypted blobs |
| pkcs11-proxy Unix socket | Infrastructure | Shared volume in OpenBao pod | **High** — MITM on this socket = intercept all PKCS#11 calls (encrypt/decrypt unseal keys) |
| softhsm-kube gRPC endpoint | Infrastructure | ClusterIP Service :5696 | **High** — any pod that can reach this Service can attempt PKCS#11 operations |
| vault-kubernetes-kms Unix socket | Infrastructure | hostPath on control-plane node | **High** — MITM = intercept all KMS encrypt/decrypt for etcd |
| softhsm-kube PV (token store) | Infrastructure | PersistentVolume on Kind node | **Critical** — contains the HSM key material in SoftHSM encrypted format. Physical access + SoftHSM PIN = key extraction |

## 2. Threat Actors

| Actor | Motivation | Access Level | Relevance to FD |
|-------|-----------|--------------|-----------------|
| Compromised pod on the cluster | Reach softhsm-kube gRPC to perform unauthorized crypto ops | Pod-level network access | Any pod that can reach softhsm-kube Service can call encrypt/decrypt — no authentication on gRPC by default |
| Compromised pkcs11-proxy sidecar | Intercept PKCS#11 calls, log plaintext unseal keys during decrypt | Container-level in OpenBao pod | The proxy sees plaintext unseal keys after HSM decrypt — it's the most sensitive position in the chain |
| Attacker with etcd dump + Transit key | Decrypt all K8s Secrets | Offline access to etcd snapshot + Vault access | Double encryption mitigates: needs both Transit key AND HSM key. But if Transit key alone is compromised, etcd Secrets are readable |
| Cluster admin with node access | Read softhsm-kube PV, extract SoftHSM tokens, brute-force PIN | Node filesystem access | SoftHSM stores keys in encrypted files — but PIN is often short/simple. Physical access to the PV = offline attack on the PIN |
| Supply chain (compromised softhsm-kube image) | Backdoor the HSM — exfiltrate keys, use weak RNG | Container runtime | The HSM image IS the trust root. A compromised image makes all crypto operations untrustworthy |
| vault-kubernetes-kms process compromise | Exfiltrate Transit tokens or cached DEKs | Process memory access | The KMS plugin holds a Vault token and may cache DEKs — process dump exposes them |

## 3. STRIDE Analysis

### softhsm-kube Pod

| Threat | Category | Risk | Mitigation |
|--------|----------|------|------------|
| Any pod on the cluster can reach softhsm-kube gRPC Service — no authentication | **S** (Spoofing) | **Critical** | Implement mTLS on gRPC (Envoy sidecar or built-in TLS). NetworkPolicy: only allow ingress from OpenBao pods (label selector). BATS: verify non-OpenBao pods cannot connect. |
| Attacker calls `GenerateKey` or `DestroyKey` on the HSM — creates rogue keys or deletes the wrapping key | **T** (Tampering) | **Critical** | gRPC server must require authentication (mTLS client cert or token). Rate limit destructive operations. BATS: verify unauthenticated `DestroyKey` fails. |
| No audit trail of HSM operations — who encrypted/decrypted what, when | **R** (Repudiation) | High | gRPC interceptor logging: every operation logged with timestamp, key ID, caller IP, operation type. BATS: verify logs contain operation entries. |
| SoftHSM PIN stored as env var — visible in pod spec, `kubectl describe pod` | **I** (Information Disclosure) | **Critical** | Store PIN in a K8s Secret, mount as file (not env var). Restrict `get pods` RBAC in softhsm-system. BATS: verify PIN is NOT in pod env vars (check pod spec). |
| softhsm-kube pod is down → Bank-Vaults cannot decrypt unseal keys → OpenBao cannot unseal → all tenants down | **D** (Denial of Service) | **Critical** | softhsm-kube is a **single point of failure** for ALL tenant unseals. Must have liveness/readiness probes, PDB, and consider HA (2 replicas with shared or replicated token store). BATS: kill softhsm-kube pod → verify it restarts and unseal recovers. |
| softhsm-kube runs as non-root with restricted PSA — no privilege escalation path | **E** (Elevation of Privilege) | Low | Already mitigated by deploy manifests (runAsNonRoot, readOnlyRootFilesystem, drop ALL capabilities). Verify in BATS. |

### pkcs11-proxy Sidecar (in OpenBao pod)

| Threat | Category | Risk | Mitigation |
|--------|----------|------|------------|
| pkcs11-proxy is a MITM position — it sees plaintext unseal keys during decrypt operations | **I** (Information Disclosure) | **Critical** | This is an architectural trade-off — the proxy MUST see the plaintext to forward it. Mitigation: use a trusted, pinned image for pkcs11-proxy. Monitor its logs for anomalies. In production, consider direct PKCS#11 library in the same container (no proxy) if the HSM is local. |
| Compromised proxy modifies PKCS#11 calls — returns wrong data or weak ciphertext | **T** (Tampering) | **Critical** | Pin proxy image by digest. Integrity check: after unseal, verify OpenBao is functional (the unseal keys were decrypted correctly). If proxy tampered with decrypt, unseal fails — fail-safe. |
| Proxy Unix socket accessible to other containers in the same pod | **E** (Elevation of Privilege) | High | The OpenBao pod has 3+ containers (vault, bank-vaults, prometheus-exporter). Only bank-vaults should access the PKCS#11 socket. Mount the socket volume only in the bank-vaults container. |
| No threats identified | **S** (Spoofing) | — | Proxy communicates over Unix socket (local) and gRPC to a known Service. No identity to spoof. |
| No threats identified | **R** (Repudiation) | — | PKCS#11 calls are logged by softhsm-kube (see above). |
| No threats identified | **D** (Denial of Service) | — | Proxy failure = bank-vaults sidecar retries. Same lifecycle as the OpenBao pod. |

### OpenBao Transit Engine

| Threat | Category | Risk | Mitigation |
|--------|----------|------|------------|

BATS: verify `vault read transit/export/encryption-key/etcd-kms` fails with 400. |

| No threats identified | **S** (Spoofing) | — | Vault token required for Transit operations. |

Must use a dedicated, HA OpenBao for KMS (not a tenant instance). For the lab: accepted single-tenant risk. |

## 4. Recommendations for SDDs

1. **SDD-001** (softhsm-kube deployment): Deploy with NetworkPolicy restricting ingress to OpenBao pods only (label selector `app.kubernetes.io/name=vault`). Store HSM PIN in K8s Secret mounted as file, NOT env var. Plan for HA (replica or PDB). BATS: verify non-OpenBao pods cannot reach softhsm-kube :5696. Verify PIN is not in pod env spec.

2. **SDD-002** (Bank-Vaults HSM unseal): Pin pkcs11-proxy image by tag/digest. Mount the PKCS#11 Unix socket volume ONLY in the bank-vaults container (not vault or prometheus-exporter). After unseal, verify OpenBao is functional (fail-safe against proxy tampering). BATS: verify unseal keys Secret is NOT plaintext base64. Verify the proxy socket is only accessible to bank-vaults.

3. **SDD-003** (Transit + vault-kubernetes-kms): Transit key `etcd-kms` must have `exportable=false` (default). KMS plugin must use a long-lived token or AppRole with auto-renewal. Socket permissions 0600. **Critical design decision**: for the lab, Transit on the test tenant's OpenBao is acceptable. For production, document that a dedicated platform-level OpenBao should serve KMS (not tenant instances). BATS: verify `vault read transit/export/...` fails. Verify KMS provider is first in encryption-config. Verify etcdctl raw read shows encrypted blob.

4. **SDD-004** (BATS tests): Must include these security-critical tests:
   - softhsm-kube NetworkPolicy: curl from non-OpenBao pod to softhsm:5696 → connection refused/timeout
   - HSM PIN not in pod env vars: `kubectl get pod -o jsonpath='{.spec.containers[*].env}'` excludes PIN
   - Unseal keys Secret is NOT plaintext: decode Secret, verify it's not valid JSON unseal keys
   - Kill softhsm-kube → next OpenBao restart → unseal fails (HSM dependency proven)
   - Restart softhsm-kube → unseal recovers
   - Transit key not exportable: `vault read transit/export/encryption-key/etcd-kms` → error
   - KMS plugin healthy: kube-apiserver logs show no KMS errors
   - Raw etcd Secret is encrypted: `etcdctl get /registry/secrets/default/test-secret` → not plaintext
   - KMS provider is first in encryption-config (before identity)

5. **SDD-005** (Integration wiring): Document the single-point-of-failure chain: softhsm-kube → pkcs11-proxy → bank-vaults → OpenBao unseal. If any link breaks, ALL tenants are affected. Production mitigation: HA softhsm, HA OpenBao, dedicated KMS OpenBao. Include a "Failure Scenarios" section in the README: what happens when each component is down.
