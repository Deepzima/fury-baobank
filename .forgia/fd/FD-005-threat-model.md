# Threat Model: FD-005 — PKI/CA engine for Kubernetes certificate management

> Generated by `/fd-threat-model` on 2026-04-18.
> Based on: `.forgia/fd/FD-005-pki-ca-engine-k8s-certs.md`

## 1. Assets

| Asset | Type | Location | Sensitivity |
|-------|------|----------|-------------|
| Root CA private key | Credential | OpenBao PKI engine (`pki-root/`) internal storage | **Critical** — compromise = ALL certs issued under this CA are untrustworthy. Total PKI compromise. |
| Intermediate CA private key | Credential | OpenBao PKI engine (`pki-k8s/`) internal storage | **Critical** — compromise = all K8s component certs untrustworthy (but Root can revoke + reissue Intermediate) |
| Root CA certificate (public) | Data | OpenBao + distributed to trust stores | **Low** — public by design; leaking it doesn't compromise security |
| Issued leaf certificates (apiserver, kubelet, etcd, front-proxy) | Data | Vault API response → consumer | **Medium** — contain public key + metadata; private key is the sensitive part |
| Issued leaf private keys | Credential | Vault API response → consumer | **High** — compromise of apiserver key = MITM on all K8s API traffic |
| CRL (Certificate Revocation List) | Data | OpenBao endpoint `/v1/pki-k8s/crl` | **Medium** — tampering with CRL could hide revoked certs |
| PKI role configurations | Data | OpenBao PKI engine config | **High** — defines what SANs/CNs are allowed; misconfiguration = certs for wrong identities |
| Vault root token (for PKI admin) | Credential | Unseal keys Secret | **Critical** — can create/modify/delete CA, roles, and issue arbitrary certs |

## 2. Threat Actors

| Actor | Motivation | Access Level | Relevance to FD |
|-------|-----------|--------------|-----------------|
| Compromised tenant OpenBao | Issue arbitrary certs for any domain/SAN | Full Vault access | If the tenant's OpenBao is compromised, the attacker can issue K8s certs for any hostname — becomes a rogue CA |
| Tenant user with Vault access | Issue certs with unauthorized SANs (e.g., certs for another tenant's apiserver) | Vault token with PKI role access | PKI roles must constrain allowed_domains and enforce_hostnames — otherwise tenant can issue certs for arbitrary SANs |
| Network attacker intercepting cert issuance | Capture private keys during API response | Network access (Docker bridge) | Without TLS on Vault API, private keys are transmitted in plaintext |
| CRL endpoint consumer | Trust a revoked cert because CRL is stale or tampered | HTTP access to CRL endpoint | CRL is served over HTTP (no auth) — a MITM could serve a fake empty CRL |
| Supply chain (compromised OpenBao) | Generate weak keys or backdoored certs | OpenBao binary | PKI key generation happens inside OpenBao — a compromised binary could use weak randomness |

## 3. STRIDE Analysis

### Root CA (pki-root/)

| Threat | Category | Risk | Mitigation |
|--------|----------|------|------------|
| Root CA private key exposed via Vault API (e.g., `vault read pki-root/cert/ca` returns cert but `pki-root/key` would return key if accessible) | **I** (Information Disclosure) | **Critical** | OpenBao does NOT expose the internal Root CA private key via API (by design for `type: internal`). Verify with BATS: `vault read pki-root/key` should fail. |
| Tenant user with root token creates a second Root CA or replaces the existing one | **T** (Tampering) | **Critical** | In the per-tenant model, the tenant owns their Root CA — this is by design (it's their PKI). For a SaaS platform, the Root CA creation should be restricted to the platform operator. BATS: verify the PKI root mount requires root token (not regular role token). |
| Root CA used to directly issue leaf certs (bypassing Intermediate) | **E** (Elevation of Privilege) | High | Do NOT create roles on `pki-root/`. Only use it to sign the Intermediate CSR. BATS: verify `vault list pki-root/roles` returns empty. |
| No threats identified | **S** (Spoofing) | — | Root CA identity is established by its certificate; cannot be spoofed without the private key. |
| No threats identified | **R** (Repudiation) | — | Vault audit logs all PKI operations. |
| No threats identified | **D** (Denial of Service) | — | Root CA is rarely used (only to sign Intermediates). |

### Intermediate CA (pki-k8s/)

| Threat | Category | Risk | Mitigation |
|--------|----------|------|------------|
| Intermediate CA issues certs with unauthorized SANs (e.g., `*.google.com`) if roles are misconfigured | **E** (Elevation of Privilege) | **Critical** | Each role must set `allowed_domains`, `enforce_hostnames=true`, `allow_bare_domains=false`, `allow_subdomains` only where needed. BATS: try to issue a cert with SAN `evil.com` via k8s-apiserver role — must fail. |
| Intermediate CA cert expires and is not renewed — all new cert issuance fails | **D** (Denial of Service) | High | Set Intermediate TTL to 1 year (lab) or 90 days (prod) with monitoring. BATS: verify Intermediate cert has expected TTL. |
| Intermediate CA private key extracted via export endpoint | **I** (Information Disclosure) | High | OpenBao internal CAs do not expose private keys. Verify: `vault read pki-k8s/key` fails. |
| An attacker issues a cert with long TTL (years) that persists after a compromise is discovered | **T** (Tampering) | High | Set `max_ttl` on each role (24h for leaf certs). BATS: verify issued cert TTL matches expected (24h), attempt to issue with TTL > max_ttl fails. |
| No threats identified | **S** (Spoofing) | — | Intermediate identity is established by Root CA signature. |
| No threats identified | **R** (Repudiation) | — | All issuance logged in Vault audit. |

### PKI Roles (k8s-apiserver, k8s-kubelet, k8s-etcd, k8s-front-proxy)

| Threat | Category | Risk | Mitigation |
|--------|----------|------|------------|
| k8s-apiserver role allows wildcard SANs — attacker issues cert for arbitrary hostname | **E** (Elevation of Privilege) | **Critical** | Set `allowed_domains=["kubernetes", "kubernetes.default", "kubernetes.default.svc", "kubernetes.default.svc.cluster.local"]`, `allow_ip_sans=true` (for 10.96.0.1), `enforce_hostnames=true`. BATS: issue cert, verify SANs are exactly the allowed ones. |
| k8s-kubelet role allows arbitrary CN — attacker issues cert as `system:node:admin` | **S** (Spoofing) | High | Set `allowed_domains=["system:node"]`, `allow_bare_domains=true`, `organization=["system:nodes"]`. BATS: verify CN starts with `system:node:` and O = `system:nodes`. |
| k8s-etcd role missing client auth EKU — etcd rejects cert | **D** (Denial of Service) | Medium | Set `server_flag=true`, `client_flag=true` on etcd role. BATS: verify EKU includes both Server Auth and Client Auth. |
| k8s-front-proxy role has server auth EKU (should be client-only) — apiserver rejects it | **T** (Tampering) | Medium | Set `server_flag=false`, `client_flag=true`. BATS: verify EKU is Client Auth only. |
| No threats identified for Information Disclosure | **I** (Information Disclosure) | — | Role configuration is not sensitive data. |
| No threats identified for Repudiation | **R** (Repudiation) | — | All cert issuance logged. |

| Threat | Category | Risk | Mitigation |
|--------|----------|------|------------|

| No threats identified | **S** (Spoofing) | — | CRL is signed by the Intermediate CA. |
| No threats identified | **E** (Elevation of Privilege) | — | CRL is read-only from consumer perspective. |
| No threats identified | **R** (Repudiation) | — | Revocation is logged in Vault audit. |

## 4. Recommendations for SDDs

1. **SDD-001** (PKI engine setup): Root CA must be `type: internal` (key never leaves OpenBao). Do NOT create roles on `pki-root/` — only use it to sign the Intermediate CSR. Set Intermediate TTL to 1 year. BATS: verify `vault list pki-root/roles` is empty, verify Intermediate chain validates against Root.

2. **SDD-002** (PKI roles): Every role must set `enforce_hostnames=true`, `max_ttl=24h`, `key_bits >= 2048`. k8s-apiserver: explicit `allowed_domains` list (no wildcards). k8s-kubelet: `organization=["system:nodes"]`. k8s-etcd: `server_flag=true, client_flag=true`. k8s-front-proxy: `server_flag=false, client_flag=true`. BATS: for each role, attempt to issue a cert with unauthorized SAN — must fail.

3. **SDD-003** (Cert issuance BATS): Verify SANs match exactly (no extra SANs). Verify Key Usage and Extended Key Usage per cert type. Verify TTL <= 24h. Verify key size >= 2048. Verify chain: `openssl verify -CAfile root.pem -untrusted intermediate.pem leaf.pem` succeeds.

4. **SDD-004** (Revocation BATS): After revoking, fetch CRL within 5s and verify serial is listed. Attempt to issue cert with revoked serial's CN — should succeed (revocation doesn't prevent reissuance, it invalidates existing cert). Verify CRL is DER-encoded and parseable.

5. **SDD-005** (Integration wiring): Document plaintext-on-wire risk for lab. Document that production requires TLS on OpenBao listener + HTTPS CRL distribution point. Include a `## Production Considerations` section in the scenario README.
